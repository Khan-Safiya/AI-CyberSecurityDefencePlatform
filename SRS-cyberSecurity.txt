# SOFTWARE REQUIREMENTS SPECIFICATION
# Multi-Agent Cyber Defence Simulation Platform
# Java Spring Boot Microservices Production-Grade System

You are building a production-grade Multi-Agent Cyber Defence Simulation Platform.

The system must be implemented using Java 21+, Spring Boot, Spring Cloud-style microservice architecture, PostgreSQL, Redis, Kafka or RabbitMQ, Docker, Docker Compose, OpenAPI, Micrometer, Prometheus, Grafana, JUnit 5, Testcontainers, and Maven or Gradle.

The platform must simulate a safe, controlled cyber defence environment with two teams:

1. Red Team
   - Performs safe, scoped, non-destructive, simulated security testing.
   - Can test built-in vulnerable sandbox services.
   - Can test verified third-party staging targets only when explicitly registered and authorized.
   - Must never attack real external systems without registration and verification.
   - Must never execute real malware, destructive payloads, credential theft, persistence, or arbitrary system commands.

2. Blue Team
   - Detects findings.
   - Prioritizes vulnerabilities.
   - Applies remediation.
   - Verifies fixes.
   - Improves defensive rules.
   - Participates in an iterative loop until the target reaches an acceptable security state.

The final system must run locally using:

docker compose up --build

It must seed a demo scenario and allow a complete end-to-end simulation.

==================================================
13. ITERATIVE SIMULATION ENGINE
==================================================

The simulation must run in iterative rounds.

Each round:

1. Red team performs scoped testing.
2. Findings are created.
3. Detection engine creates detection events.
4. Blue team analyzes findings.
5. Blue team proposes remediation.
6. Remediation service applies fixes when possible.
7. Verification service validates fixes.
8. Red team retests.
9. Scoring service updates scores.
10. Event-log-service records the timeline.

Stopping conditions:

- No new vulnerabilities found for N consecutive rounds.
- All critical and high vulnerabilities are fixed.
- Maximum rounds reached.
- Maximum duration reached.
- Manual stop requested.
- Unsafe action detected.
- Target becomes unavailable.
- Policy violation occurs.

Configurable fields:

- maxRounds
- maxDurationMinutes
- stopWhenNoNewFindingsForRounds
- minimumAcceptedRiskLevel
- retestEnabled
- retestDelaySeconds

The orchestrator must never allow an infinite loop.
